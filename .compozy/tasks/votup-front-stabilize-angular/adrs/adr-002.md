# ADR-002: Simplified BaseComponent with inject() and Generics

## Status
Accepted

## Date
2026-05-16

## Context
The current `BaseComponent<T>` uses a complex pattern: it receives an `Injector` via constructor, creates dynamic `InjectionToken` instances at runtime to obtain `BaseService<T>` instances, and manually calls `injector.get()` for all dependencies (HttpClient, Router, FormBuilder, ActivatedRoute). This creates 10+ lint errors per component and makes dependencies opaque.

Three approaches were considered:
1. Eliminate BaseComponent entirely — use composition (helper functions, signal mixins) instead of inheritance
2. Keep a simplified BaseComponent — refactor with `inject()` and generics, remove manual Injector pattern
3. Convert BaseComponent logic into an injectable service — components use composition via `inject(CrudHelper)`

## Decision
Keep a simplified BaseComponent class that uses `inject()` for all dependencies and proper generics for type safety. The class retains its utility methods (CRUD, pagination, navigation, form management) but eliminates the manual `Injector` and dynamic `InjectionToken` pattern.

## Alternatives Considered

### Alternative 1: Eliminate BaseComponent Entirely
- **Description**: Remove the base class; each component manages its own services via `inject()` and shares logic via composition
- **Pros**: Maximum decoupling, no inheritance baggage, each component is self-contained
- **Cons**: Massive refactoring effort for 13+ extending components, significant code duplication for CRUD/pagination patterns, high risk of regressions
- **Why rejected**: The existing components heavily depend on BaseComponent's CRUD, pagination, and form lifecycle methods. Removing it would require rewriting every component from scratch.

### Alternative 2: Convert to Injectable Service
- **Description**: Move BaseComponent logic into a `CrudHelper` or `PageManager` service that components inject
- **Cons**: Services cannot directly manage component lifecycle (FormGroup, ViewChild, ngOnDestroy), would require awkward workarounds, state management becomes fragmented
- **Why rejected**: BaseComponent manages component-specific concerns (forms, route params, pagination UI state) that don't map cleanly to a service pattern.

## Consequences

### Positive
- `inject()` makes all dependencies explicit and visible at the class level
- Proper generics eliminate `any` from service and form interactions
- Minimal disruption to existing components — they continue extending a base class
- Lint errors from `prefer-inject` and `Injector` pattern are eliminated

### Negative
- Inheritance still couples components to the base class
- Some components may not need all BaseComponent features, receiving unnecessary baggage

### Risks
- Dynamic `InjectionToken` pattern for creating additional `BaseService<K>` instances (used by components that need multiple services) must be replaced with a cleaner approach, such as direct `inject(BaseService)` with factory providers
