# ADR-003: Functional HTTP Interceptor for Authentication

## Status
Accepted

## Date
2026-05-16

## Context
The current `AuthInterceptor` implements the legacy `HttpInterceptor` class interface with constructor injection. Angular 21 provides `withInterceptors()` for functional interceptors that use `inject()` internally. The class-based `HttpInterceptor` interface is deprecated in favor of the functional approach.

## Decision
Convert the `AuthInterceptor` to a functional interceptor (`authInterceptorFn`) and register it via `withInterceptors([authInterceptorFn])` in the `provideHttpClient()` call during bootstrap.

## Alternatives Considered

### Alternative 1: Keep Class with inject() Migration
- **Description**: Migrate to `inject()` but keep the `HttpInterceptor` class interface
- **Pros**: Minimal code change
- **Cons**: Uses deprecated API, doesn't align with standalone-first architecture, `withInterceptorsFromDi()` would still be needed
- **Why rejected**: The functional interceptor is the Angular 21 standard and eliminates the need for `withInterceptorsFromDi()`, simplifying the HTTP client configuration.

## Consequences

### Positive
- Eliminates `withInterceptorsFromDi()` from bootstrap configuration
- Fully aligned with Angular 21 standalone APIs
- Simpler to test and compose with other interceptors
- One less class-based service with constructor injection to migrate

### Negative
- Functional interceptors cannot store state between requests (not a concern for auth headers and error handling)

### Risks
- None significant — the conversion is straightforward and the logic (add Accept-Language header, handle 401 logout, display error toasts) maps directly to a function
