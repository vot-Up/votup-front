# ADR-005: Functional Route Guards

## Status
Accepted

## Date
2026-05-16

## Context
The current `AppGuard` implements `CanActivate`, `CanLoad`, and `CanDeactivate` as a class with constructor injection. Angular 21 prefers functional guards (`CanActivateFn`, `CanLoadFn`) that use `inject()` internally. Class-based guards are legacy.

## Decision
Convert `AppGuard` to functional guard equivalents (`authGuardFn` for `canActivate`, `canLoad`) using `inject()` for `AuthService` and `Router`. The `CanDeactivate` logic (currently a passthrough returning `true`) will be removed or converted to a simple functional guard if needed.

## Alternatives Considered

### Alternative 1: Keep Class with inject()
- **Description**: Migrate to `inject()` but keep the class-based guard interface
- **Pros**: Minimal change
- **Cons**: Class-based guards are legacy, route configs reference classes differently than functions, doesn't align with Angular 21 patterns
- **Why rejected**: Functional guards are simpler, more testable, and are the Angular 21 standard. The current CanDeactivate logic is a no-op passthrough.

## Consequences

### Positive
- Route configuration uses function references directly: `canActivate: [authGuard]`
- Cleaner, stateless guard implementation
- Better tree-shaking — unused guard code can be eliminated
- Eliminates the need to provide `AppGuard` in module/provider arrays

### Negative
- None significant for this codebase — the guard logic is simple (check token, redirect to login)

### Risks
- The `CanDeactivate` check currently always returns `true` — it should be removed or properly implemented if navigation guards are needed in the future
