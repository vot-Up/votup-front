# ADR-001: Incremental Schematics-First Migration Strategy

## Status
Accepted

## Date
2026-05-16

## Context
The votup-front project runs Angular 21.2 but still uses the legacy NgModule-based architecture with 97 lint errors (23 prefer-standalone, 74 prefer-inject) and 51 warnings (no-explicit-any). The team wants to modernize the codebase to fully adopt Angular 21 patterns (standalone components, signals, inject(), zoneless) to improve readability, maintainability, and long-term developer experience. There are no production constraints or time limitations blocking a thorough migration.

Three migration approaches were considered:
1. Incremental Schematics-First — run Angular schematics in sequence with validation checkpoints, then refactor structural code
2. Bottom-Up by Layer — manually refactor each layer (models → services → base → components) completely before moving up
3. Big-Bang Schematics + Immediate Cleanup — run all schematics at once followed by a single large cleanup pass

## Decision
Adopt the Incremental Schematics-First Migration Strategy (Approach 1).

Execute Angular's official migration schematics in sequential phases with build/lint validation between each phase, followed by targeted structural refactoring and zoneless adoption. This leverages maximum automation while maintaining safe, verifiable progress.

## Alternatives Considered

### Alternative 1: Bottom-Up by Layer
- **Description**: Manually refactor each architectural layer completely (models → services → base → components) before moving to the next layer
- **Pros**: Cleanest intermediate state, no mixed legacy/modern code, full control over each change
- **Cons**: Significantly slower, does not leverage automated schematics, highly manual and error-prone, risks introducing bugs without validated tooling
- **Why rejected**: Does not take advantage of Angular's battle-tested migration schematics, resulting in much more manual work for equivalent outcomes

### Alternative 2: Big-Bang Schematics + Immediate Cleanup
- **Description**: Run all migration schematics in a single batch, then perform one large cleanup pass covering typing, base classes, zoneless, and signals all at once
- **Pros**: Fastest total timeline, single change window
- **Cons**: High risk — difficult to isolate problems when something breaks, hard to review, harder to debug regressions, no intermediate validation points
- **Why rejected**: The lack of validation checkpoints makes it risky and harder to troubleshoot, especially with a codebase that has structural patterns like BaseComponent with dynamic InjectionTokens

## Consequences

### Positive
- Maximum use of automated tooling reduces manual effort and error surface
- Each phase produces a buildable, lint-validatable codebase — regressions are caught early
- Incremental approach allows the team to stop at any phase and still have a functional, improved codebase
- Schematics are maintained by the Angular team and handle edge cases that manual migration might miss
- Natural checkpoint structure fits well with phased rollout planning

### Negative
- Intermediate phases produce mixed legacy/modern code (e.g., standalone components still using BaseComponent with Injector)
- Requires discipline to continue through all phases — risk of stopping after schematics without completing structural refactoring
- Schematics may reformat code, requiring lint/format cleanup after each run

### Risks
- Schematics may not handle BaseComponent's dynamic InjectionToken pattern correctly, requiring manual fixes
- The project has zero tests, so regressions can only be caught via build verification and manual testing
- Mitigation: validate build + lint after each schematic run; consider adding minimal smoke tests early in the process

## Implementation Notes
- Run schematics in order: standalone (3 sub-steps) → inject → control-flow → signal-input → output → signal-queries
- After schematics complete, refactor BaseComponent and BaseService with proper typing
- Adopt signals for local state and computed values
- Remove Zone.js and enable zoneless change detection as the final phase
- Each phase must pass `ng build` and `ng lint` before proceeding
