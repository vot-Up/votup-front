# ADR-004: Zoneless with Zone.js Fallback Strategy

## Status
Accepted

## Date
2026-05-16

## Context
Angular 21 supports zoneless change detection by default for new projects, but existing applications that depend on Zone.js need a migration path. The votup-front application currently uses Zone.js and many third-party components (ng-zorro-antd) may still depend on it for change detection.

## Decision
Adopt zoneless change detection using `provideExperimentalZoneless()` in the bootstrap configuration, while keeping Zone.js available as a fallback during the transition. Remove Zone.js completely only after validating that all async operations, HTTP calls, router navigation, and ng-zorro-antd interactive components work correctly without it.

## Alternatives Considered

### Alternative 1: Immediate Full Zoneless
- **Description**: Remove Zone.js from polyfills and dependencies immediately, use `provideExperimentalZoneless()` with no fallback
- **Pros**: Cleanest result, smallest bundle from the start
- **Cons**: High risk — ng-zorro-antd may not fully support zoneless yet, async operations might break silently, no recovery path if issues are found late
- **Why rejected**: Zero test coverage makes immediate full zoneless too risky. The fallback strategy allows incremental validation.

## Consequences

### Positive
- Gradual validation reduces risk of hard-to-debug change detection failures
- Zone.js can be removed once confidence is established
- Signals-based change detection still works during the transition period
- Allows testing ng-zorro-antd compatibility under zoneless without commitment

### Negative
- Bundle still includes Zone.js during transition
- Two change detection mechanisms active simultaneously (Zone.js + signals) may cause double change detection cycles
- Must explicitly remove Zone.js and fallback in a later step

### Risks
- ng-zorro-antd v21.2.2 may have components that rely on Zone.js for internal change detection — mitigation: test each interactive component type systematically
